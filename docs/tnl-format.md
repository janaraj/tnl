# TNL File Format Specification

Authoritative reference for the TNL file format, as implemented by the shipped parser, resolver, and verifier in this repository.

This document describes what the parser actually accepts. When the implementation and this document disagree, the implementation wins and this document is updated to match.

Source of truth for the format itself: [`src/parser.ts`](../src/parser.ts). Classification rules: [`src/resolver.ts`](../src/resolver.ts). Verification tiers: [`src/verifier.ts`](../src/verifier.ts).

---

## Overview

A **TNL file** (`tnl/<id>.tnl`) is a structured English contract describing one behavioral surface in a codebase — a CLI command, an MCP tool, a route handler, a workflow, etc. It is plain text with three zones:

- **Machine zone** — key/value metadata the tool reads.
- **Contract zone** — numbered clauses with RFC 2119 keywords and optional annotation prefixes.
- **Human zone** — free prose: `intent`, `non-goals`, `rationale`.

TNL files are authored by an agent, approved by a human, and then used as the target for implementation. The sidecar file (`tnl/.resolved/<id>.meta.json`) is a cached hash of the unit and each clause, plus the classifier's verdict. The verifier checks structural invariants (paths exist, dependencies resolve, declared tests still exist).

---

## File layout

```
tnl/
├── workflow.tnl               # one scope: repo-wide unit
├── <feature>.tnl              # one scope: feature unit per file
├── <another-feature>.tnl
├── .resolved/
│   ├── <feature>.meta.json    # sidecar, regenerated by `tnl resolve`
│   └── ...
└── .staging/                  # transient MCP staging area (gitignored)
    └── <diff-id>.json
```

Each `.tnl` file contains exactly one unit. The filename stem (everything before `.tnl`) MUST match the file's `id:` field — the parser rejects a mismatch.

Files are read as UTF-8. CR/LF and LF line endings are both accepted (CRLF is normalized to LF before parsing). Trailing whitespace on each line is ignored. `#` starts a line comment; the remainder of the line is discarded.

---

## Machine zone fields

The machine zone is the contiguous block of top-level `key: value` lines at the top of the file, before any section starts.

| Field | Required | Type | Notes |
|---|---|---|---|
| `id` | yes | kebab-case string | MUST match `^[a-z][a-z0-9-]*$`. MUST equal the filename stem. |
| `title` | yes | non-empty string | Short human label. |
| `scope` | yes | `repo-wide` \| `feature` | See [Scope](#scope). |
| `owners` | yes | bracket list of `@handle`s | MUST be non-empty. |
| `paths` | required when `scope: feature`; forbidden when `scope: repo-wide` | bracket list of repo-relative file paths | Non-empty when present. |
| `surfaces` | optional | bracket list of strings | Named external surfaces (CLI subcommand, MCP tool, route). Purely descriptive. |
| `dependencies` | optional | bracket list of TNL ids | Each id MUST resolve to `tnl/<id>.tnl`. Verifier checks existence. |

Bracket-list syntax is simple and literal:

```
owners: [@jana, @alice]
paths: [src/foo.ts, src/bar.ts]
dependencies: [parser, resolve]
```

Commas separate entries; whitespace around entries is trimmed. No quoting — commas and brackets are reserved characters inside a value. Empty lists (`[]`) are only legal for `surfaces` and `dependencies`.

Unknown machine-zone fields are a parse error. Duplicate fields are a parse error.

## Contract zone (behaviors)

The `behaviors:` section is a list of clauses. Each clause is a bullet (`- ...`), optionally continued on indented subsequent lines. At least one clause is required — an empty `behaviors:` section is a parse error.

```
behaviors:
  - The API MUST return 401 when the Authorization header is absent.
  - [semantic] The error response body SHOULD be compact and not leak stack traces.
  - [test: tests/api.test.ts::rate_limit_blocks_after_60] The rate limiter MUST return 429 on the 61st request in any 60-second window.
  - The service MAY emit structured logs for each blocked request.
```

### RFC 2119 keywords

Each clause is classified by the keyword it contains. Supported keywords:

- **MUST** / **MUST NOT** — hard requirement. Non-satisfaction is a failure.
- **SHOULD** / **SHOULD NOT** — strong preference. Deviation requires stated reason.
- **MAY** — permission. The clause records an allowed option, not an obligation.

### Clause classes

The resolver assigns each clause to one of four classes. Class drives downstream handling by the verifier and by the hook injection.

| Class | Trigger | Verification |
|---|---|---|
| `structural` | clause has MUST / MUST NOT and no `[semantic]` or `[test:]` prefix | No automated check at tier 1-2; documented invariant. |
| `test-backed` | clause has a `[test: <file>::<name>]` prefix | Tier-2 verifier checks the test exists in the named file (literal-string search, no execution). |
| `semantic` | clause has a `[semantic]` prefix | No automated tier-1-2 check; reserved for tier-3 (deferred). |
| `advisory` | clause has SHOULD / SHOULD NOT / MAY and no `[semantic]` or `[test:]` prefix | No automated check; records a non-blocking preference. |

A clause without any RFC 2119 keyword AND without `[semantic]` or `[test:]` is a resolve error — the classifier refuses to assign it to any class.

---

## Clause classification

Two optional prefixes may appear at the start of a clause, after the `- ` bullet marker:

### `[semantic]` prefix

Marks clauses whose verification requires judgment rather than a structural check. Tier 1-2 verification skips these (reports them as `unchecked` in `tnl verify` output). Tier 3 (deferred) would run an LLM against `[semantic]` clauses.

```
- [semantic] The middleware SHOULD log user IDs consistently without leaking request bodies.
```

### `[test: <file>::<name>]` prefix

Binds the clause to a named test in a named file. The verifier opens the file and searches for the literal name as a substring; if found, the clause passes tier 2. If the file is absent or the name is missing from the file, the clause fails.

```
- [test: tests/api.test.ts::returns_429_on_exceeded] The rate limiter MUST return 429 on the 61st request.
```

The file path is repo-relative. The test name is whatever literal appears in the source — typically the string passed to `it(...)` / `test(...)` / `describe(...)`. The verifier does not execute the test; it just checks the name appears in the file.

### Mutual exclusion

`[semantic]` and `[test:]` are mutually exclusive — a clause carrying both is a parse error. If a clause genuinely needs both paths (a structural test AND a judgment-based invariant), split it into two clauses.

## Human zone

Three sections, all free-form markdown:

| Section | Required | Shape |
|---|---|---|
| `intent:` | yes | Prose paragraph. One paragraph is canonical. |
| `non-goals:` | optional | Bullet list. Each bullet is an explicit scope fence. |
| `rationale:` | optional | Prose paragraph(s). Tradeoffs, gotchas, "why" behind choices. |

Sections are introduced by the key followed by a colon on its own line, with the body indented below. Inline values after the colon are a parse error — the parser expects an indented body.

```
intent:
  One-paragraph plain-English description.

non-goals:
  - An explicit scope fence.
  - Another thing this TNL deliberately doesn't cover.

rationale:
  Why we chose this approach over alternatives.
```

## Scope

Two values are legal:

- **`scope: feature`** — the TNL governs specific files (required `paths:` field) and surfaces. Applies when the code being edited overlaps with the declared paths.
- **`scope: repo-wide`** — the TNL governs cross-cutting behavior with no specific files. Applies to every session. `paths:` is forbidden; the impacted-TNL detector always returns repo-wide units regardless of which files are touched.

This repository's `tnl/workflow.tnl` is the canonical repo-wide example: its clauses apply to every coding session regardless of what's being built.

---

## Dependencies

`dependencies: [other-id-1, other-id-2]` declares that this TNL references clauses or behaviors defined in other TNLs. The verifier's tier-1 `dependencies-resolve` check confirms each named id exists as `tnl/<id>.tnl`.

Dependencies are informational — they don't automatically pull in the referenced unit's clauses, and there is no transitive resolution. Their purpose is to make cross-unit coupling visible and auditable.

Common uses:

- A feature TNL that amends another (e.g. `gemini-mcp-register` depends on `mcp-auto-register` for shape).
- A feature TNL that builds on an abstraction (e.g. `cli-diff` depends on `cli-skeleton`).

---

## Sidecar

After a successful `tnl resolve` run (triggered explicitly by `tnl resolve <id>`, or implicitly on `approve_tnl_diff`), a sidecar file is written at `tnl/.resolved/<id>.meta.json` with this shape:

```json
{
  "unit_hash": "a3f1b2c4d5e6...",
  "resolved_at": "2026-04-22T14:30:45.123Z",
  "clauses": {
    "L-1": { "hash": "1a2b3c...", "class": "structural" },
    "L-2": { "hash": "4d5e6f...", "class": "test-backed", "test": { "file": "tests/rate_limit.test.ts", "name": "returns_429_on_exceeded" } },
    "L-3": { "hash": "7a8b9c...", "class": "semantic" },
    "L-4": { "hash": "0d1e2f...", "class": "advisory" }
  }
}
```

Fields:

- **`unit_hash`** — SHA-256 of the entire `.tnl` source (after line-ending normalization + trailing-whitespace stripping).
- **`resolved_at`** — ISO-8601 timestamp of when the sidecar was generated. **Preserved byte-for-byte across runs when the `unit_hash` is unchanged** — re-running `tnl resolve` on an unchanged file does not update the timestamp, so sidecars produce a zero-line `git diff`.
- **`clauses`** — object keyed by clause id (`L-1`, `L-2`, …, where `N` is the 1-based index in the `behaviors:` list — not the source line number).
  - **`hash`** — SHA-256 of the clause's normalized text (whitespace collapsed, trimmed).
  - **`class`** — one of `structural`, `test-backed`, `semantic`, `advisory`.
  - **`test`** — present only on `test-backed` clauses. Object with `file` (repo-relative path) and `name` (test name literal).

The sidecar is checked into git alongside the `.tnl` source. It lets tools detect "did this clause change?" cheaply without re-parsing.

## Verification tiers

`tnl verify` runs per-unit checks and reports a typed result line per check.

### Tier 1 — structural contract checks

- **`paths-exist`** — every entry in the `paths:` field exists on disk. Fails if any path is missing.
- **`dependencies-resolve`** — every id in `dependencies:` has a `tnl/<id>.tnl` file. Fails if any id is dangling.

### Tier 2 — test-binding integrity

For each `test-backed` clause:

- Open the file named in `[test: <file>::<name>]`.
- Search the file for the literal `<name>` as a substring.
- Pass if found; fail if the file is missing or the name is absent.

The verifier does **not** execute the test. CI runs tests via the existing test runner (Vitest, pytest, etc.); `tnl verify` just checks that declared test bindings haven't gone stale.

### Tier 3 — semantic (deferred)

`semantic` clauses are reported as `unchecked` at tier 1-2. A tier-3 verifier — an LLM-based judge running at CI time — is deferred post-v1. Whether it ships, and in what form, is TBD.

### Exit codes

- `0` — all checks passed (or unchecked).
- `2` — one or more checks failed.

### CI integration

`tnl init` writes `.github/workflows/tnl-verify.yml` which runs `npx -y typed-nl verify` on every push and pull request. Failure fails the workflow.

## Edit-vs-new guidance

A new TNL file is justified only for:

- A genuinely new behavioral surface (new CLI subcommand, new MCP tool, new subsystem).
- A cross-cutting policy that spans multiple existing TNLs (`workflow.tnl` is the canonical example).
- A feature with a clear boundary not already covered by any existing TNL.

Any other change — modifying inputs, outputs, semantics, validation, constraints, or identity rules of an existing surface — is an edit to the existing file. **Do not create a new TNL to patch an existing one.** Property changes to existing surfaces (new validation rule, tightened constraint, new input/output on an existing route) are edits.

## Examples

### `scope: feature` — a CLI subcommand

```tnl
id: hello
title: Sample hello subcommand
scope: feature
owners: [@example]
paths: [src/commands/hello.ts]
surfaces: [tnl hello]

intent:
  `tnl hello` prints a greeting. Exists as a minimal example of the
  TNL format for documentation purposes.

behaviors:
  - `src/commands/hello.ts` MUST register a `hello` command on the default command registry.
  - When invoked with no arguments, the command MUST print `Hello, world!` to stdout and exit 0.
  - [test: tests/hello.test.ts::greets_with_name] When invoked with `--name <value>`, the command MUST print `Hello, <value>!`.
  - [semantic] The greeting SHOULD avoid exclamation marks in contexts where enthusiasm is culturally inappropriate.

non-goals:
  - Internationalization of the greeting.
  - Configurable templates for the output.

rationale:
  Example only — present in this spec to demonstrate a realistic
  `scope: feature` unit with all four clause classes represented.
```

### `scope: repo-wide` — cross-cutting policy

```tnl
id: no-secrets
title: No secrets in committed code
scope: repo-wide
owners: [@security]

intent:
  Secrets (API keys, credentials, private certificates) MUST NOT
  be committed to the repository. Sample repo-wide policy.

behaviors:
  - The repository MUST NOT contain files named `.env`, `.env.local`, or `credentials.json` under version control.
  - [test: tests/gitignore.test.ts::blocks_env_files] The `.gitignore` MUST exclude `.env*` and `credentials.json`.
  - When a contributor discovers a committed secret, the response SHOULD be to rotate the secret first, then rewrite history.
  - [semantic] Secrets-in-logs are as dangerous as secrets-in-source; logging pipelines MUST NOT persist raw authentication headers.

non-goals:
  - Enforcement tooling (pre-commit hooks, git-secrets) — landed separately.
  - Cryptographic standards for stored secrets — covered by a different policy.

rationale:
  Example only — demonstrates a `scope: repo-wide` unit (no `paths:`
  field) with clauses that apply across the entire codebase.
```

Both examples parse via [`parseTnl()`](../src/parser.ts). The spec-verification test in [`tests/docs/format-spec.test.ts`](../tests/docs/format-spec.test.ts) extracts every fenced `tnl` code block from this document and asserts each one parses without error — so if the parser ever tightens a check, the stale example in this doc fails the test and forces the fix.
